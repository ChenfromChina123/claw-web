## 1. 项目概述

**Claw-Web** 是一个基于 AI 的代码助手平台，提供多用户隔离的 AI 对话、代码执行、终端访问等能力。项目采用 **Master-Worker 容器化架构**，通过 Docker 容器实现用户级别的资源隔离和弹性伸缩。

### 核心能力

| 能力 | 描述 |
|------|------|
| AI 对话 | 支持通义千问、Claude 等多模型对话 |
| 代码执行 | 通过 PTY 终端提供安全的代码执行环境 |
| 多 Agent 协作 | 内置多种 Agent 类型，支持团队编排 |
| MCP 工具协议 | 支持外部工具服务器接入 |
| 多租户隔离 | 用户级别的容器隔离与资源配额 |
| 实时通信 | WebSocket 双向通信，支持流式响应 |

## 3. Master-Worker 容器化架构

### 3.1 架构设计理念

Claw-Web 采用 **Master-Worker 架构**，核心思想是：

- **Master 容器**：负责全局管控（认证、路由、调度），不处理业务逻辑
- **Worker 容器**：负责业务执行（会话、工具、Agent），每个容器服务一个用户
- **热容器池**：预启动空闲容器，实现毫秒级分配

### 3.2 请求流转流程

```
用户请求 → Nginx(8888) → Master(3000)
                              │
                              ├─ 1. JWT 认证验证
                              ├─ 2. 查找用户→容器映射
                              ├─ 3. 无映射？触发调度策略
                              │     ├─ 热池获取空闲容器
                              │     └─ 热池为空？创建新容器
                              ├─ 4. 代理请求到 Worker 容器
                              │     ├─ 设置 X-User-Id 头
                              │     ├─ 设置 X-User-Admin 头
                              │     └─ 设置 X-Proxy-Origin 头
                              └─ 5. 返回 Worker 响应
```

### 3.3 容器角色对比

| 特性 | Master 容器 | Worker 容器 |
|------|------------|-------------|
| **角色标识** | `CONTAINER_ROLE=master` | `CONTAINER_ROLE=worker` |
| **认证** | 验证 JWT Token | 信任 Master 传递的用户信息 |
| **容器编排** | 初始化编排器 | 跳过编排器初始化 |
| **最大用户数** | 不限（仅路由） | `MAX_USERS=1` |
| **内存限制** | 768MB | 256MB-2048MB（按等级） |
| **CPU 限制** | 1.5 核 | 0.5-4 核（按等级） |
| **端口** | 3000（固定） | 3100-3200（动态分配） |
| **Docker Socket** | 挂载（管理容器） | 不挂载 |

### 3.4 Worker 容器认证机制

Worker 容器**完全不参与认证**，通过以下机制信任 Master：

```typescript
// Worker 模式下的认证逻辑
if (CONTAINER_ROLE === 'worker') {
  // 检查请求是否来自 Master
  if (request.headers.get('X-Proxy-Origin') === 'claw-web-master') {
    // 从头部提取用户信息
    return {
      userId: request.headers.get('X-User-Id'),
      isAdmin: request.headers.get('X-User-Admin') === 'true'
    }
  }
  // 非 Master 请求直接拒绝
  return { userId: null, isAdmin: null }
}
```

async function handleRequest(req) {

### 4.4 请求代理机制

Master 代理请求到 Worker 的核心逻辑：

```typescript
async function proxyToWorkerContainer(req, containerName, path, userInfo) {
  const targetUrl = `http://${containerName}:3000${path}`
  
  const headers = new Headers(req.headers)
  headers.set('X-Forwarded-For', 'claw-web-master')
  headers.set('X-Proxy-Origin', 'claw-web-master')
  headers.set('X-User-Id', userInfo.userId)
  headers.set('X-User-Admin', userInfo.isAdmin ? 'true' : 'false')
  
  return await fetch(targetUrl, { method, headers, body })
}
```

## 5. WebSocket 实时通信系统

### 5.1 架构概览

WebSocket 系统由两层组成：

1. **wsMessageRouter**：高层消息路由，处理业务消息
2. **wsBridge (WebSocketManager)**：底层 RPC 桥接，处理 RPC 调用

### 5.2 消息类型

| 类型 | 方向 | 描述 |
|------|------|------|
| `ping/pong` | 双向 | 心跳检测 |
| `login` | 客户端→服务端 | Token 登录 |
| `register` | 客户端→服务端 | 用户注册 |
| `create_session` | 客户端→服务端 | 创建会话 |
| `load_session` | 客户端→服务端 | 加载会话 |
| `list_sessions` | 客户端→服务端 | 列出会话 |
| `user_message` | 客户端→服务端 | 发送用户消息 |
| `interrupt_generation` | 客户端→服务端 | 中断生成 |

[schedulingPolicy.ts](file:///d:/Users/Administrator/AistudyProject/HAHA/claw-web/server/src/orchestrator/schedulingPolicy.ts) 实现了基于用户等级的调度策略：

**用户等级与配额：**

| 等级 | 优先级 | 最大会话 | 存储配额 | 空闲超时 | 独占容器 |
|------|--------|---------|---------|---------|---------|
| VIP | 1 | 20 | 2000MB | 永不回收 | ✅ |
| Premium | 2 | 15 | 1000MB | 30分钟 | ❌ |
| Regular | 3 | 10 | 500MB | 5分钟 | ❌ |
| Trial | 4 | 3 | 200MB | 2分钟 | ❌ |

**调度流程：**

```
scheduleContainer(userId, userData)
  ├── 1. 确定用户等级 (determineUserTier)
  ├── 2. 检查系统资源状况
  ├── 3. 选择调度策略
  │     ├── VIP → scheduleForVIP (独占容器)
  │     ├── 有空闲容器 → scheduleFromPool
  │     ├── 内存充足 → scheduleNewContainer
  │     └── 资源紧张 → handleResourceExhaustion
  │           ├── VIP → 排队等待
  │           ├── Premium → 回收资源后重试
  │           ├── Regular → 排队等待
  │           └── Trial → 直接拒绝
  └── 4. 返回调度结果
```
