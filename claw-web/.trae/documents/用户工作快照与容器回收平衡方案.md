# 用户工作快照与容器回收平衡方案

## 一、问题分析

### 1.1 当前机制

| 组件     | 当前配置                | 说明                          |
| ------ | ------------------- | --------------------------- |
| 容器空闲超时 | 5分钟                 | 热池容器空闲5分钟后被销毁               |
| 快照机制   | AgentStatusSnapshot | 内存中的执行状态快照                  |
| 工作区存储  | 共享Volume            | `/app/workspaces` 挂载到Master |

### 1.2 存在的问题

1. **快照不持久化**：AgentStatusSnapshot 存储在内存中，容器销毁后丢失
2. **工作进度丢失**：用户正在执行的任务状态无法恢复
3. **用户体验差**：长时间操作（如编译大型项目）可能被中断

### 1.3 核心矛盾

```
用户期望 ──────────────────────▶ 资源效率
     │                                    │
     ▼                                    ▼
工作不丢失 ◀──── 快照 + 持久化 ────▶ 容器及时回收
```

## 二、解决方案

### 2.1 分层快照策略

| 快照类型      | 触发时机          | 存储位置         | 保留时间 | 恢复能力    |
| --------- | ------------- | ------------ | ---- | ------- |
| **实时快照**  | 每个工具调用完成      | Redis + 数据库  | 24小时 | 可恢复到任意点 |
| **检查点快照** | 关键操作完成（编译/测试） | 数据库          | 7天   | 可恢复     |
| **最终快照**  | 会话结束/容器回收前    | 数据库 + Volume | 永久   | 完全恢复    |

### 2.2 容器回收时机

#### 2.2.1 回收触发条件（需同时满足）

```
条件A：容器空闲超时 OR
条件B：用户会话结束

AND

条件C：最终快照已保存成功
```

#### 2.2.2 不同用户等级的超时配置

| 用户等级       | 空闲超时 | 快照频率 | 最大并发容器 |
| ---------- | ---- | ---- | ------ |
| ADMIN      | 永不回收 | 实时   | 5      |
| ENTERPRISE | 30分钟 | 5分钟  | 3      |
| PRO        | 10分钟 | 2分钟  | 2      |
| BASIC      | 5分钟  | 1分钟  | 1      |
| FREE       | 2分钟  | 30秒  | 1      |

### 2.3 工作区保护机制

#### 2.3.1 快照内容

```typescript
interface WorkSnapshot {
  id: string
  userId: string
  sessionId: string
  containerId: string
  createdAt: Date

  // 文件系统状态
  workspaceFiles: {
    path: string
    checksum: string
    modifiedAt: Date
  }[]

  // 进程状态
  runningProcesses: {
    pid: number
    command: string
    startedAt: Date
  }[]

  // 执行状态
  executionState: {
    currentWorkingDirectory: string
    activeTasks: Task[]
    recentHistory: ToolCall[]
  }

  // Git状态（如有）
  gitState: {
    branch: string
    uncommittedFiles: string[]
    lastCommit: string
  }
}
```

#### 2.3.2 恢复策略

```
容器被回收
    │
    ▼
检查最终快照 ────▶ 存在? ──▶ 是 ──▶ 恢复工作区到新容器
    │                              │
    │                              ▼
    │                         通知用户
    │
    ▼
不存在快照 ────▶ 通知用户数据可能丢失
```

## 三、实施方案

### 阶段一：基础设施（1-2天）

1. **创建快照服务** `WorkSnapshotService`

   * 文件快照：tar + gzip 压缩工作目录

   * 数据库表：`work_snapshots`

   * Redis缓存：最近快照索引

2. **数据库Schema**

   ```sql
   CREATE TABLE work_snapshots (
     id VARCHAR(36) PRIMARY KEY,
     user_id VARCHAR(36) NOT NULL,
     session_id VARCHAR(36) NOT NULL,
     container_id VARCHAR(64),
     snapshot_type ENUM(' realtime', 'checkpoint', 'final') NOT NULL,
     workspace_path VARCHAR(512),
     file_checksums JSON,
     process_state JSON,
     execution_state JSON,
     git_state JSON,
     created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
     expires_at TIMESTAMP,
     INDEX idx_user_session (user_id, session_id),
     INDEX idx_expires (expires_at)
   );
   ```

### 阶段二：集成回收流程（1-2天）

1. **修改 ContainerOrchestrator**

   * 回收前触发快照保存

   * 验证快照保存成功

   * 记录容器生命周期日志

2. **添加生命周期钩子**

   ```typescript
   interface ContainerLifecycleHook {
     beforeRecycle(containerId: string): Promise<WorkSnapshot>
     afterRecover(containerId: string, snapshot: WorkSnapshot): Promise<void>
     onSnapshotFailed(error: Error): void
   }
   ```

### 阶段三：用户界面（1天）

1. **快照管理面板**

   * 查看快照历史

   * 手动创建快照

   * 一键恢复

2. **通知系统**

   * 快照保存成功/失败通知

   * 容器回收提醒（可选）

### 阶段四：优化（持续）

1. **增量快照**

   * 仅保存变更的文件

   * 使用 rsync 算法

2. **后台压缩**

   * 使用更多CPU时间换IO

3. **快照清理策略**

   * 实时快照：24小时

   * 检查点快照：7天

   * 最终快照：永久（用户可手动删除）

## 四、配置项

| 环境变量                           | 默认值   | 说明            |
| ------------------------------ | ----- | ------------- |
| `SNAPSHOT_ENABLED`             | true  | 是否启用快照        |
| `SNAPSHOT_REALTIME_INTERVAL`   | 60000 | 实时快照间隔（毫秒）    |
| `SNAPSHOT_COMPRESSION_LEVEL`   | 6     | gzip压缩级别（1-9） |
| `SNAPSHOT_RETENTION_DAYS`      | 7     | 快照保留天数        |
| `WORKSPACE_PROTECTION_ENABLED` | true  | 工作区保护         |
| `CONTAINER_GRACEFUL_TIMEOUT`   | 30000 | 优雅停止超时（毫秒）    |

## 五、预期效果

| 指标    | 优化前  | 优化后    |
| ----- | ---- | ------ |
| 工作丢失率 | \~5% | \~0.1% |
| 容器回收率 | 70%  | 85%    |
| 用户满意度 | 75%  | 95%    |
| 存储成本  | 基准   | +15%   |

## 六、风险与缓解

| 风险   | 概率 | 影响 | 缓解措施            |
| ---- | -- | -- | --------------- |
| 快照失败 | 低  | 高  | 回收前双重验证，失败则保留容器 |
| 存储溢出 | 中  | 中  | 快照压缩 + 自动清理     |
| 恢复失败 | 低  | 中  | 提供手动恢复选项        |
