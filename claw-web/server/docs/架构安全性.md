## 二、职责边界

### Master 职责（中心化）

| 模块 | 职责 | 不做 |
|------|------|------|
| **用户认证** | JWT 签发与验证 | Agent 执行 |
| **会话管理** | 会话 CRUD、消息存储 | 文件操作 |
| **数据库访问** | 唯一数据库连接点 | 业务逻辑 |
| **容器调度** | 创建/分配/回收容器 | 终端操作 |
| **权限控制** | 用户等级，配额检查 | 工具执行 |
| **会话上下文** | 加载历史消息、构建上下文 | LLM 调用 |
| **流式转发** | 透明中转，不解析内容 | - |

### Worker 职责（纯沙箱）

| 模块 | 职责 | 不做 |
|------|------|------|
| **Agent 执行** | 调用 LLM、执行工具 | 会话保存 |
| **工具调用** | 文件读写、终端命令 | 用户认证 |
| **PTY 终端** | 交互式 shell | 数据库写入 |
| **流式响应** | 实时推送 AI 输出 | 权限验证 |

### 架构优势

| 改进 | 效果 |
|------|------|
| **Worker 无数据库连接** | 用户无法通过终端访问/修改数据库 |
| **所有写操作在 Master** | 会话数据一致性由 Master 事务保证 |
| **工具权限由 Master 控制** | 用户等级决定的工具限制在 Master 层统一执行 |
| **内部 API 令牌认证** | Worker 只响应 Master 的信任请求 |

## 四、核心代码设计

### 4.1 Master: Agent 执行 API

**新建文件**: `server/src/server/agentApi.ts`

```typescript
/**
 * Agent 执行 API - Master 专用
 * 负责会话上下文管理和结果持久化
 */

import { SessionManager } from '../services/sessionManager'
import { getAgentExecutor } from '../agents/executor'

const sessionManager = SessionManager.getInstance()

/**
 * 在 Worker 上执行 Agent - 流式版本
 * Master 负责加载上下文，Worker 只负责执行
 * 使用 pipeThrough 实现透明中转
 */
export async function executeAgentOnWorker(
  workerUrl: string,
  userId: string,
  sessionId: string,
  message: string,
  sendEvent: (event: string, data: unknown) => void
): Promise<{ success: boolean; error?: string }> {
  
  // 1. Master 加载会话上下文
  const sessionData = await sessionManager.loadSession(sessionId)
  if (!sessionData) {
    return { success: false, error: 'Session not found' }
  }

  // 2. 构建 Agent 执行请求
  const requestBody = {
    userId,
    sessionId,
    message,
    context: {
      messages: sessionData.messages,
      tools: getAgentTools(),
      quota: await getUserQuota(userId),
    },
    stream: true,
  }

  // 3. 调用 Worker 执行
  const response = await fetch(`${workerUrl}/api/internal/agent/execute`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Master-Token': process.env.MASTER_INTERNAL_TOKEN!,
    },
    body: JSON.stringify(requestBody),
  })

  if (!response.ok) {
    return { success: false, error: `Worker error: ${response.status}` }
  }

  // 4. 流式转发 - 使用 pipeThrough 透明中转
  // 策略一：Master 转发"不等待"，避免阻塞
  const stream = response.body!
    .pipeThrough(new TextDecoderStream())
    .pipeThrough(new TransformStream({
      transform(chunk, controller) {
        // 收到一行，立刻转发，不解析内容
        controller.enqueue(chunk)
      }
    }))

  // 5. 异步处理工具调用结果 (策略二)
  // 注意：这需要在 TransformStream 外部单独处理
  // 因为 SSE 格式需要完整解析才能识别 event type
  
  return { success: true }
}

/**
 * 异步保存工具调用结果 - Fire and Forget
 * 不阻塞流式输出
 */
async function saveToolCallAsync(sessionId: string, toolCall: ToolCall): Promise<void> {
  try {
    await sessionManager.addToolCall(sessionId, toolCall)
  } catch (error) {
    console.error('[AgentAPI] 异步保存工具调用失败:', error)
    // 保存失败不影响主流程
  }
}
```

```typescript
export async function startServer(): Promise<void> {
  const containerRole = process.env.CONTAINER_ROLE || 'master'

  if (containerRole === 'worker') {
    console.log('[Worker] Starting in sandbox mode...')
    await startWorkerMode()

  }

  console.log('[Master] Starting in master mode...')
  await initDatabase()
  await initAuth()
  await initContainerOrchestrator()
}

/**
 * Worker 启动：只初始化执行引擎
 */
async function startWorkerMode(): Promise<void> {
  const server = Bun.serve({
    port: 3000,
    hostname: '0.0.0.0',

    async fetch(req) {
      const url = new URL(req.url)
      const path = url.pathname

      // ========== Worker 内部 API ==========

      // 1. 健康检查
      if (path === '/api/health') {
        return json({ status: 'healthy', role: 'worker' })
      }

      // 2. Agent 执行 (主入口)
      if (path === '/api/internal/agent/execute' && req.method === 'POST') {
        const masterToken = req.headers.get('X-Master-Token')
        if (masterToken !== process.env.MASTER_INTERNAL_TOKEN) {
          return json({ error: 'Unauthorized' }, { status: 401 })
        }

        const body = await req.json()
        return await handleAgentExecute(body)
      }

      // 3. PTY 创建
      if (path === '/api/internal/pty/create' && req.method === 'POST') {
        return await handlePtyCreate(req)
      }

      // 4. 文件操作
      if (path.startsWith('/api/internal/files/')) {
        return await handleFileOperation(req, path)
      }

      // 5. 工具执行
      if (path === '/api/internal/tools/execute' && req.method === 'POST') {
        return await handleToolExecute(req)
      }

      return json({ error: 'Not found' }, { status: 404 })
    },
  })
}

/**
 * Agent 执行处理 - SSE 流式响应
 */
async function handleAgentExecute(body: {
  userId: string
  sessionId: string
  message: string
  context: {
    messages: Message[]
    tools: Tool[]
    quota: UserQuota
  }
}): Promise<Response> {
  const { userId, sessionId, message, context } = body

  const stream = new ReadableStream({
    async start(controller) {
      const sendEvent = (type: string, data: unknown) => {
        // SSE 格式：每个事件以 \n\n 分隔
        const payload = `data: ${JSON.stringify({ type, data })}\n\n`
        controller.enqueue(payload)
      }

      try {
        const executor = getAgentExecutor()
        await executor.execute({
          userId,
          sessionId,
          message,
          messages: context.messages,
          tools: context.tools,
          quota: context.quota,
          onEvent: sendEvent,
        })
      } catch (error) {
        sendEvent('error', { message: error.message })
      } finally {
        controller.close()
      }
    },
  })

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
    },
  })
}
```

```typescript
        await tx.insert('tool_calls', toolCall)
      }

        updated_at: new Date(),
      })
    })
  }
```

## 五、流式响应性能评价

### 5.1 流式数据路径

```
LLM → Worker → Master → Client
        ↑
        └── Docker 内部网络: 0.5ms - 2ms RTT
```

### 5.2 性能指标

| 维度 | 评价 | 影响 |
|------|------|------|
| **首字时间 (TTFB)** | **极佳** | Master 收到第一个 Chunk 后立即转发，不等待后续 |
| **流稳定性** | **优秀** | Master 维持长连接，即便 Worker 重启，Master 可以尝试重连 |
| **吞吐量** | **良好** | 瓶颈主要在 LLM 生成速度，而非 Master 转发 |

### 5.3 延迟分析

| 阶段 | 延迟 | 说明 |
|------|------|------|
| LLM 生成 | 20-50 chars/s | 主要瓶颈 |
| Docker 内部网络 | 0.5-2ms | Master ↔ Worker RTT |
| Master 转发 | <1ms | 使用 Stream Pipe，不解析 JSON |
| 人类感知 | 0 | 文本流人类完全无法察觉 |

### 5.4 核心架构优势

#### A. 响应速度的"反直觉"优化

虽然多了一层中转，但 Worker 变得 **极轻（Stateless）**：
- **启动速度快**：不需要等待 MySQL 连接初始化、不需要加载复杂的权限模型
- **冷启动缩短**：当有新用户进来，Worker 容器能以毫秒级速度进入工作状态

#### B. 安全与性能的平衡

- **Master 挡箭牌**：用户的恶意代码在 Worker 里跑，最多把 Worker 搞死，Master 依然可以通过数据库保存的快照瞬间恢复环境
- **数据库保护**：避免了 N 个容器同时竞争数据库连接（Connection Exhaustion），这通常是导致 IDE 卡顿的隐形杀手

## 六、优化策略

### 策略一：Master 转发"不等待"

使用 `pipeThrough` 实现流式管道转发，**不等待 JSON 解析完整**。

```typescript
// ✅ 正确做法：水管接头模式，O(1) 转发
const response = await fetch(workerUrl, options)
const stream = response.body!
  .pipeThrough(new TextDecoderStream())
  .pipeThrough(new TransformStream({
    transform(chunk, controller) {
      // 收到一行，立刻转发
      // 不解析内容，不阻塞
      controller.enqueue(chunk)
    }
  }))

return new Response(stream, {
  headers: { 'Content-Type': 'text/event-stream' }
})

// ❌ 错误做法：等待完整 JSON
const full = await response.json()  // 阻塞！
processAndReturn(full)
```

### 策略二：异步保存（Fire and Forget）

当 Master 检测到"聊天内容" Chunk 时，先发给用户，**后台异步写入数据库**。

```typescript
// 事件处理器
async function handleSSEEvent(event: SSEEvent) {
  // 立即转发给用户（不阻塞）
  sendToClient(event)

  if (event.type === 'message_complete') {

  }
```

## 七、流式传输细节优化

### 7.1 TCP 粘包问题处理

在 `Master → Client` 的 HTTP 流中，如果网络波动，多个 SSE 事件可能会挤在一个数据包里。需要正确处理 `\n\n` 分隔符。

```typescript
/**
 * SSE 事件解析器
 * 正确处理 TCP 粘包问题
 */
class SSEParser {
  private buffer: string = ''
  private eventBuffer: Map<string, string> = new Map()

  /**
   * 解析 SSE 数据块
   * @param chunk 新收到的数据块
   * @returns 解析出的事件数组
   */
  parse(chunk: string): SSEParsedEvent[] {
    this.buffer += chunk
    const events: SSEParsedEvent[] = []
    
    // 按 SSE 事件分隔符分割 (\n\n)
    const eventBlocks = this.buffer.split(/\n\n/)
    
    // 最后一个块可能是不完整的，保留到下次处理
    this.buffer = eventBlocks.pop() || ''
    
    for (const block of eventBlocks) {
      const event = this.parseEventBlock(block)
      if (event) {
        events.push(event)
      }
    }
    
    return events
  }
  
  private parseEventBlock(block: string): SSEParsedEvent | null {
    let type = 'message'  // 默认事件类型
    let data = ''
    
    for (const line of block.split('\n')) {
      if (line.startsWith('event:')) {
        type = line.slice(6).trim()
      } else if (line.startsWith('data:')) {
        data += line.slice(5).trim() + '\n'
      }
    }
    
    if (!data) return null
    
    try {
      const payload = JSON.parse(data)
      return { type, data: payload }
    } catch {
      return null
    }
  }
}

/**
 * Master 流式转发器
 * 包含 SSE 解析和异步保存
 */
class StreamForwarder {
  private parser = new SSEParser()
  
  async forward(
    workerStream: ReadableStream<Uint8Array>,
    sendToClient: (event: string, data: unknown) => void
  ) {
    const reader = workerStream.getReader()
    
    try {
      while (true) {
        const { done, value } = await reader.read()
        if (done) break
        
        const chunk = new TextDecoder().decode(value)
        
        // 解析 SSE 事件
        const events = this.parser.parse(chunk)
        
        for (const event of events) {
          // 立即转发
          sendToClient(event.type, event.data)
          
          // 异步保存（不阻塞）
          this.asyncPersist(event)
        }
      }
    } finally {
      reader.releaseLock()
    }
  }
  
  private asyncPersist(event: SSEParsedEvent) {
    if (event.type === 'tool_result') {
      setImmediate(() => saveToolCallAsync(sessionId, event.data))
    }
  }
}
```

### 7.2 PTY 终端 vs Agent 对话的流量区分

Agent 的流式输出（纯文本）和 PTY 终端的流式输出（带 ANSI 颜色转义字符）是不一样的，需要区分处理。

| 流量类型 | 协议 | 特点 |
|----------|------|------|
| **Agent 对话** | SSE (Server-Sent Events) | 结构化事件，可解析 JSON |
| **PTY 终端** | Binary WebSocket | 字节流，含 ANSI 转义字符 |

```typescript
/**
 * Master 流量路由
 */
async function handleWebSocketMessage(ws: WebSocket, message: WSMessage) {
  switch (message.type) {
    case 'user_message':
      // Agent 对话：走 SSE 转发
      await forwardAgentStream(ws, message)
      break
      
    case 'pty_input':
      // PTY 终端：直接 Binary WebSocket 转发
      await forwardPTYStream(ws, message)
      break
      
    case 'ping':
      // 心跳：直接响应
      ws.send(JSON.stringify({ type: 'pong' }))
      break
  }
}

/**
 * Agent 流式转发 - SSE
 */
async function forwardAgentStream(ws: WebSocket, message: UserMessage) {
  const response = await fetch(`${workerUrl}/api/internal/agent/execute`, {
    method: 'POST',
    body: JSON.stringify({ /* ... */ })
  })
  
  // 使用 TransformStream 透明转发
  response.body!
    .pipeThrough(new TextDecoderStream())
    .pipeTo(new WritableStream({
      write(chunk) {
        ws.send(chunk)  // 直接转发 SSE 文本
      }
    }))
}

/**
 * PTY 流式转发 - Binary WebSocket
 * 不解析内容，直接透传
 */
async function forwardPTYStream(ws: WebSocket, message: PtyInput) {
  // PTY 输入直接转发到 Worker
  await fetch(`${workerUrl}/api/internal/pty/input`, {
    method: 'POST',
    body: JSON.stringify(message)
  })
  
  // PTY 输出从 Worker WebSocket 接收
  const ptyWs = await connectToWorkerWS(`${workerUrl}/api/internal/pty/stream`)
  
  ptyWs.onmessage = (event) => {
    ws.send(event.data)  // 直接透传二进制
  }
}
```

### 7.4 Nginx 卸载无状态路由（进阶优化）

对于纯文件流下载等无状态路由，可以通过 Nginx 的 `auth_request` 模块直接卸载到 Worker，让 Master 只负责"点头"。

```nginx
# nginx.conf

upstream master_backend {
    server master:3000;
}

upstream worker_backend {
    server worker-pool:3000;
}

server {
    listen 80;
    
    # Agent 对话 - 必须经过 Master (需要认证、会话管理)
    location /api/agent/ {
        auth_request /auth-verify;
        proxy_pass http://master_backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        
        # SSE 支持
        proxy_buffering off;
        proxy_cache off;
    }
    
    # 纯文件下载 - 可以直接卸载到 Worker
    location /api/files/download/ {
        auth_request /auth-verify;  # 只验证，不经过 Master
        proxy_pass http://worker_backend;
    }
    
    # 健康检查 - 直接到 Worker
    location /health {
        proxy_pass http://worker_backend;
    }
    
    # 认证验证 (Nginx 内部子请求)
    location = /auth-verify {
        internal;
        proxy_pass http://master_backend/api/auth/verify;
        proxy_method POST;
        proxy_pass_request_body off;
        proxy_set_header Content-Length "";
        proxy_set_header X-Original-URI $request_uri;
    }
}
```

### 7.5 宿主机目录挂载（Volume Mount）

这是目前最简单且性能最强的文件持久化方案。Master 所在的宿主机就是天然的"存储池"。

#### 核心逻辑

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        存储架构                                          │
└─────────────────────────────────────────────────────────────────────────┘

                    宿主机文件系统
                    /data/claws/workspaces/
                    ├── user-abc123/
                    │   ├── project-001/  ←── 用户工作目录
                    │   │   ├── src/
                    │   │   ├── package.json
                    │   │   └── README.md
                    │   └── project-002/
                    │       └── ...
                    └── user-xyz789/
                        └── ...

                            │
                            │ Bind Mount
                            ▼
                    Worker Container
                    /workspace  ──→  宿主机 /data/claws/workspaces/{userId}/
```

#### 路径设计

| 层级 | 路径 | 说明 |
|------|------|------|
| 根目录 | `/data/claws/workspaces/` | 宿主机统一存储根目录 |
| 用户目录 | `/data/claws/workspaces/{userId}/` | 按用户 ID 隔离 |
| 项目目录 | `/data/claws/workspaces/{userId}/{projectId}/` | 每个项目独立子目录 |

#### Docker 启动逻辑

```typescript
// containerOrchestrator.ts - 宿主机目录挂载
async function createWorkerContainer(userId: string, projectId: string) {
  // 构建宿主机路径
  const hostWorkspacePath = `/data/claws/workspaces/${userId}/${projectId}`
  const containerWorkspacePath = '/workspace'

  // 确保宿主机目录存在
  await fs.mkdir(hostWorkspacePath, { recursive: true })

  const dockerCmd = [
    'docker run -d',
    `--name worker-${userId}-${Date.now()}`,
    
    // 目录挂载：将宿主机目录映射到容器内
    '-v', `${hostWorkspacePath}:${containerWorkspacePath}`,
    
    // 其他配置...
    '--network clawex-net',
    '-m 512m',
    '--pids-limit 100',
    
    'worker-image'
  ].join(' ')

  await execAsync(dockerCmd)
  
  console.log(`[Container] 目录已挂载: ${hostWorkspacePath} → ${containerWorkspacePath}`)
}
```

#### 宿主机初始化脚本

```bash

# 设置权限（宿主机普通用户）

```

#### 方案优势

| 特性 | 效果 |
|------|------|
| **实时持久化** | Agent 执行 `touch a.txt` 瞬间，文件就写入宿主机磁盘 |
| **零同步成本** | 没有"复制"和"回传"过程 |
| **二次加载极快** | 新容器启动，挂载同一目录，0 秒恢复环境 |
| **容器故障容忍** | 容器崩溃、销毁、断电，文件不丢失 |
| **隔离性强** | 每个用户目录独立，权限最小化 |

#### 与传统方案的对比

| 方案 | 持久化 | 同步延迟 | 实现复杂度 | 推荐度 |
|------|--------|----------|------------|--------|
| **宿主机挂载** | ✅ 实时 | 0ms | 低 | ⭐⭐⭐⭐⭐ |
| Docker Volume | ✅ 实时 | 0ms | 中 | ⭐⭐⭐⭐ |
| NFS 网络存储 | ✅ 实时 | 5-20ms | 高 | ⭐⭐⭐ |
| 对象存储回传 | ⚠️ 延迟 | 100ms+ | 高 | ⭐⭐ |
| 内存存储 | ❌ 丢失 | 0ms | 低 | ❌ |

### 8.2 PTY 命令过滤

```typescript
// ptyManager.ts - 命令黑名单
class PTYManager {
  private readonly BLOCKED_COMMANDS = [
    // 容器操作
    'docker', 'podman', 'containerd',
    // 系统操作
    'sudo', 'su', 'passwd', 'useradd', 'userdel',
    // 网络扫描
    'nmap', 'tcpdump', 'wireshark',
    // 文件系统
    'mount', 'umount', 'fdisk', 'mkfs',
  ]

  validateCommand(command: string): { allowed: boolean; reason?: string } {
    for (const blocked of this.BLOCKED_COMMANDS) {
      if (command.includes(blocked)) {
        return { 
          allowed: false, 
          reason: `禁止执行: ${blocked}` 
        }
      }
    }
    return { allowed: true }
  }
}
```

## 架构总结

### 核心要点

1. **Worker = 纯沙箱**
   - 不连接数据库
   - 不处理认证
   - 只执行 Agent 和工具

2. **Master = 唯一入口**
   - 统一认证
   - 会话管理
   - 数据持久化
   - 流式转发（透明中转）

3. **通信模式**
   - Master → Worker：HTTP + 内部 Token
   - Worker → Master：SSE 流式响应（无反向调用）

### 一句话评价

> **这是一个"教科书级别"的沙箱架构方案。** 以"极高可靠性"换取"极微小延迟"，在分布式系统中是典型的生产级方案。

### 技术栈选择

使用 **Bun** 的 `ReadableStream` 和 `pipeThrough` 接口，能让流式响应像原生直连一样丝滑。

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **转发模式** | `pipeThrough` 透明中转 | O(1) 延迟，不阻塞 |
| **数据保存** | `setImmediate` 异步写入 | 不阻塞流式输出 |
| **容器预热** | 用户打开 IDE 时预分配 | 消除冷启动延迟 |
| **SSE 解析** | 自定义 Parser | 处理 TCP 粘包 |
| **PTY 协议** | Binary WebSocket | 区分 Agent 对话流量 |
