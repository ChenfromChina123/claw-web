# 移除热池机制方案

## 核心原则

**一个用户 = 一个 Worker 容器 = 一个工作空间**

热池机制在当前架构中形同虚设，应移除并改为按需创建容器。

---

## 问题分析

### 当前热池机制的问题

1. **热池容器无用户上下文** - 预启动的容器没有挂载工作空间，无法直接分配给用户
2. **每次都要销毁重建** - 用户请求后，热池容器必须销毁，再创建专用容器，等于白启动
3. **浪费资源** - 预启动的容器占用 CPU、内存和端口，但几乎立即被销毁
4. **增加复杂度** - 维护热池、健康检查、空闲清理等逻辑，但没有带来实际收益

### 为什么热池不适用本项目

热池适用于"同质化"容器（无状态服务），但 Worker 容器是"有状态"的：
- 每个容器挂载特定用户的工作空间
- 容器内有用户的环境变量和配置
- 无法将已初始化的容器直接分配给另一个用户

```typescript
// 新逻辑（正确）
async assignContainerToUserInternal(userId: string, username?: string, userTier?: UserTier, lockKey?: string) {
  // 1. 确保用户工作空间存在
  const workspaceManager = getWorkspaceManager()
  const workspace = await workspaceManager.getOrCreateUserWorkspace(userId)
  const workspacePath = workspaceManager.getActualWorkspacePath(userId)
  
  // 2. 直接为用户创建容器（不再经过热池）
  const createResult = await this.createUserContainer(userId, username, userTier, workspacePath)
  if (!createResult.success) {
    return createResult as OrchestratorResult<UserContainerMapping>
  }
  const container = createResult.data!
  
  // 3. 创建用户映射
  const mapping: UserContainerMapping = {
    userId,
    container,
    assignedAt: new Date(),
    sessionCount: 0,
    lastActivityAt: new Date()
  }
  
  this.userMappings.set(userId, mapping)
  
  // 4. 保存映射到数据库
  await this.saveUserMappingToDB(mapping)
  
  return { success: true, data: mapping }
}
```

#### 步骤 5：简化端口分配

**文件**：`server/src/master/orchestrator/containerOrchestrator.ts`

**当前逻辑**（复杂）：

```typescript
private usedPorts: Set<number> = new Set()
private portCounter: number = 0

async allocatePort(): Promise<number> {
  // 复杂的端口搜索和冲突检测
  while (this.config.basePort + this.portCounter < maxPort) {
    const port = this.config.basePort + this.portCounter
    this.portCounter++
    if (this.usedPorts.has(port)) continue
    const isAvailable = await this.isPortAvailable(port)
    if (isAvailable) {
      this.usedPorts.add(port)
      return port
    }
  }
}
```

**新逻辑**（简单）：

```typescript
/**
 * 为用户容器分配端口
 * 直接使用 Docker 动态端口映射，避免端口冲突
 */
async allocatePort(preferredPort?: number): Promise<number> {
  if (preferredPort) {
    // 如果指定了端口，检查是否可用
    const isAvailable = await this.isPortAvailable(preferredPort)
    if (isAvailable) {
      return preferredPort
    }
  }
  
  // 不指定端口，让 Docker 自动分配
  // 创建容器时使用 -p 4000（不指定宿主机端口）
  // 然后通过 docker inspect 获取分配的端口
  return 0 // 0 表示让 Docker 自动分配
}

/**
 * 检查端口是否可用
 */
private async isPortAvailable(port: number): Promise<boolean> {
  const net = require('net')
  return new Promise((resolve) => {
    const server = net.createServer()
    server.once('error', (err: any) => {
      resolve(err.code !== 'EADDRINUSE')
    })
    server.once('listening', () => {
      server.close()
      resolve(true)
    })
    server.listen(port, '0.0.0.0')
  })
}
```

**建议方案**：使用 Docker 动态端口映射

```typescript
// 创建容器时不指定宿主机端口
const dockerCmd = `docker run -d --name ${containerName} -p 4000 ...`
// Docker 会自动分配一个可用端口

// 容器创建后获取实际端口
```

**改动**：

```typescript
/**
 * 释放用户容器（当用户所有活动都结束时调用）
 * 一个用户一个容器，不区分会话
 */
async releaseUserContainer(userId: string, force: boolean = false): Promise<OrchestratorResult<void>> {
  const mapping = this.userMappings.get(userId)
  if (!mapping) {
    return { success: true } // 已经释放
  }
  
  const container = mapping.container
  
  if (force) {
    // 强制销毁容器
    await this.destroyContainer(container.containerId)
  } else {
    // 直接销毁容器（不再归还热池）
    await this.destroyContainer(container.containerId)
  }
  
  // 移除映射
  this.userMappings.delete(userId)
  await this.updateUserMappingStatusInDB(userId, 'released')
  
  console.log(`[ContainerOrchestrator] 用户 ${userId} 的容器已释放`)
  return { success: true }
}
```

**改动点**：
- 移除 `returnToWarmPool()` 调用
- 统一销毁容器（不归还热池）

```typescript
  // 检查热池容器
  for (const [containerId, container] of this.warmPool) {
    const isHealthy = await this.checkContainerHealth(containerId)

```

**新逻辑**（只检查用户容器）：

```typescript
/**
 * 启动用户容器健康检查循环
 * 定期检查用户容器是否健康，不健康时尝试恢复
 */
private startHealthCheckLoop(): void {
  const healthCheckLoop = async () => {
    try {
      // 只检查用户容器
      for (const [userId, mapping] of this.userMappings) {
        const isHealthy = await this.checkContainerHealth(mapping.container.containerId)
        if (!isHealthy) {
          console.error(`[ContainerOrchestrator] 用户 ${userId} 的容器不健康，尝试恢复...`)
          await this.recoverUserContainer(userId)
        }
      }
    } catch (error) {
      console.error('[ContainerOrchestrator] 健康检查循环出错:', error)
    } finally {
      // 定时执行
      this.healthCheckTimer = setTimeout(healthCheckLoop, this.config.healthCheckIntervalMs)
    }
  }
  
  this.healthCheckTimer = setTimeout(healthCheckLoop, this.config.healthCheckIntervalMs)
}

/**
 * 恢复用户容器
 */
private async recoverUserContainer(userId: string): Promise<void> {
  const mapping = this.userMappings.get(userId)
  if (!mapping) return
  
  // 1. 尝试重启容器
  const restarted = await this.restartUserContainer(userId)
  if (restarted) {
    console.log(`[ContainerOrchestrator] 用户 ${userId} 的容器重启成功`)
    return
  }
  
  // 2. 重启失败，重新分配容器
  console.error(`[ContainerOrchestrator] 用户 ${userId} 的容器重启失败，重新分配...`)
  await this.destroyContainer(mapping.container.containerId)
  await this.assignContainerToUser(userId, mapping.username, mapping.userTier)
}
```

## 风险评估

| 风险项 | 风险等级 | 影响 | 缓解措施 |
|--------|----------|------|----------|
| 容器创建时间增加 | 低 | 首次分配容器需要 90 秒 | 用户通常只分配一次容器 |
| 并发请求冲突 | 低 | 同一用户并发请求可能创建多个容器 | 已有分布式锁保护 |
| 端口分配冲突 | 低 | Master 重启后可能端口冲突 | 使用 Docker 动态端口映射 |

---

## 预期收益

| 指标 | 移除前 | 移除后 | 变化 |
|------|--------|--------|------|
| 代码行数 | ~2600 行 | ~1500 行 | **减少 42%** |
| 内存占用 | 热池容器（每个 512MB） | 无热池 | **节省资源** |
| 端口占用 | 预占用 3-10 个端口 | 按需占用 | **节省端口** |
| 逻辑复杂度 | 高（热池+用户容器） | 低（只有用户容器） | **显著降低** |
| 容器分配时间 | 90-120 秒（热池无用） | 90-120 秒 | **不变** |

**注意**：移除热池后，容器分配时间不变（因为热池本来就形同虚设），但代码复杂度大幅降低，资源消耗减少。

## 后续优化建议

如果需要提升容器分配速度，可以考虑以下方案（与热池无关）：

1. **优化 Worker 容器启动时间**
   - 减少 Worker 镜像大小
   - 优化 Worker 启动脚本
   - 目标：从 90 秒降到 30 秒

2. **使用容器快照**
   - 创建容器时从快照恢复，而非从零启动
   - Docker 支持容器快照功能
   - 目标：从 90 秒降到 10 秒

3. **预创建容器（有用户上下文）**
   - 提前知道哪些用户会来（如用户登录时）
   - 预创建用户容器（挂载工作空间）
   - 目标：从 90 秒降到 1 秒

但这些是后续优化，当前先移除热池，简化架构。
