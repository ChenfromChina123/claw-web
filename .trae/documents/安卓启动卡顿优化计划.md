# 安卓App启动卡顿优化计划

## 问题分析

根据代码审查，发现以下导致启动卡顿的主要原因：

### 1. 主线程阻塞问题
- **ClawCodeApplication.onCreate()** 中同步初始化大量组件
- 数据库初始化 (`AppDatabase.getInstance()`) 在主线程执行
- Repository、TokenManager、UserManager 等组件同步创建
- OkHttpClient 和 Retrofit 构建在主线程执行

### 2. 数据加载问题
- **SessionListScreen** 的 `sessionDisplayData` 计算在每次重组时执行
- `generatePreview()` 函数包含大量字符串匹配逻辑，在列表渲染时重复执行
- `formatTime()` 每次都会创建新的 SimpleDateFormat 实例

### 3. 动画和渲染问题
- **RunningIndicator** 使用 `rememberInfiniteTransition` 持续运行动画
- 会话列表项包含复杂的背景色计算和图标类型判断

## 优化方案

### 阶段1: Application异步初始化

#### 1.1 延迟初始化非关键组件
- 将 `notificationManager`、`appDatabase`、`cachedChatRepository` 改为延迟初始化
- 使用 `by lazy` 委托实现按需加载

#### 1.2 异步初始化数据库
- 使用 `Room.inMemoryDatabaseBuilder` 预加载优化
- 数据库初始化移到后台协程

#### 1.3 网络组件异步创建
- OkHttpClient 和 Retrofit 改为延迟初始化
- 首次网络请求时才真正创建

### 阶段2: 列表性能优化

#### 2.1 数据预计算优化
- 将 `sessionDisplayData` 使用 `remember` 缓存，添加合适的 key
- 使用 `derivedStateOf` 优化筛选逻辑

#### 2.2 时间格式化优化
- 使用静态 SimpleDateFormat 实例（线程安全版本）
- 避免每次重组都创建新的日期格式器

#### 2.3 预览文本优化
- 将 `generatePreview` 改为预计算或使用 remember 缓存
- 避免在每次重组时执行大量字符串匹配

### 阶段3: 动画和渲染优化

#### 3.1 列表项动画优化
- 使用 `LazyColumn` 的 `key` 参数确保复用
- 添加 `contentType` 优化重组范围

#### 3.2 运行指示器优化
- 仅在可见时运行动画
- 使用 `LaunchedEffect` 控制动画生命周期

### 阶段4: 启动流程优化

#### 4.1 启动页优化
- 添加启动主题避免白屏
- 延迟加载非必要数据直到首屏显示

#### 4.2 认证检查优化
- `AuthCheckScreen` 的认证检查改为异步执行
- 避免在 `LaunchedEffect` 中执行同步阻塞操作

## 具体修改文件

1. **ClawCodeApplication.kt** - 异步初始化组件
2. **SessionListScreen.kt** - 列表性能优化
3. **MainActivity.kt** - 启动流程优化
4. **ChatViewModel.kt** - 初始化优化

## 验证方案

1. 使用 Android Profiler 检查主线程阻塞
2. 测量启动时间（冷启动和热启动）
3. 检查帧率是否达到 60fps
4. 验证列表滚动流畅度
