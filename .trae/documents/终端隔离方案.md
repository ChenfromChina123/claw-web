# 终端隔离方案

## 一、需求分析

### 核心需求
1. **用户隔离**：每个用户只能在自己的工作目录中操作
2. **路径限制**：禁止切换到上层目录或访问其他用户目录
3. **路径隐藏**：不向用户暴露真实路径信息，仅显示用户唯一标识符
4. **虚拟化终端**：基于 Bun 原生 PTY API 实现安全的终端隔离

### 当前技术栈
- 使用 Bun 的 `Bun.Terminal` 或 `bun-pty-rust` 原生 API
- 运行 bash shell
- 需要实现沙箱隔离机制

---

## 二、可行性方案对比

### 方案 1：Linux Namespaces + Mount 隔离（推荐⭐）

#### 技术原理
使用 Linux 内核的 Namespaces 机制创建隔离环境：
- **Mount Namespace**：隔离文件系统视图
- **User Namespace**：隔离用户权限
- **PID Namespace**：隔离进程视图

#### 实现方式
```typescript
import { spawn } from 'bun-pty-rust';

// 使用 unshare 命令创建隔离的 namespace
const terminal = spawn('unshare', [
  '--mount',      // 挂载命名空间隔离
  '--user',       // 用户命名空间隔离
  '--pid',        // 进程命名空间隔离
  '--fork',       // fork 新进程
  '--map-root-user', // 映射 root 用户
  'bash'
], {
  cwd: `/app/workspaces/users/${userId}`,
  env: {
    ...process.env,
    HOME: `/app/workspaces/users/${userId}`,
  }
});
```

#### 挂载点隔离配置
```bash
# 1. 创建私有挂载命名空间
unshare --mount

# 2. 将用户目录绑定挂载为新的根
mount --bind /app/workspaces/users/${userId} /app/workspaces/users/${userId}

# 3. 设置为私有挂载，防止传播
mount --make-private /app/workspaces/users/${userId}

# 4. 使用 chroot 限制根目录（可选，增强隔离）
chroot /app/workspaces/users/${userId} /bin/bash
```

#### 优点
- ✅ 内核级隔离，安全性高
- ✅ 完全隔离文件系统视图
- ✅ 防止目录遍历攻击
- ✅ 支持多层级防护

#### 缺点
- ⚠️ 需要 root 权限或 CAP_SYS_ADMIN 能力
- ⚠️ Windows 不支持（需要 WSL 或虚拟机）

### 方案 2：命令白名单 + 路径验证中间层

#### 技术原理
在 PTY 和 shell 之间添加一个命令拦截层，验证所有命令的路径合法性。

#### 实现架构
```
用户输入 → 命令解析器 → 路径验证 → 白名单检查 → PTY → Shell
           ↓
       拒绝非法命令
```

#### 核心代码示例
```typescript
class SecureTerminal {
  private allowedCommands = ['ls', 'cd', 'pwd', 'echo', 'cat', 'mkdir', 'rm', 'cp', 'mv'];
  private userRoot: string;
  private currentPath: string;
  
  async validateCommand(cmd: string): Promise<boolean> {
    // 1. 解析命令
    const parts = cmd.trim().split(/\s+/);
    const baseCmd = parts[0];
    
    // 2. 检查命令白名单
    if (!this.allowedCommands.includes(baseCmd)) {
      return false;
    }
    
    // 3. 验证路径参数
    if (['cd', 'cat', 'ls', 'mkdir', 'rm'].includes(baseCmd)) {
      const targetPath = parts[1];
      const resolvedPath = path.resolve(this.currentPath, targetPath);
      
      // 4. 确保路径在用户根目录内
      if (!resolvedPath.startsWith(this.userRoot)) {
        return false;
      }
      
      // 5. 检查是否尝试访问上层目录
      if (targetPath.includes('..')) {
        return false;
      }
    }
    
    return true;
  }
  
  async executeCommand(cmd: string): Promise<string> {
    if (!await this.validateCommand(cmd)) {
      throw new Error('命令被拒绝：超出访问权限');
    }
    
    // 执行命令...
  }
}
```

#### 优点
- ✅ 跨平台兼容（Windows/Linux/Mac）
- ✅ 实现简单，无需特殊权限
- ✅ 灵活可控

#### 缺点
- ⚠️ 可被绕过（如通过 `bash -c` 嵌套执行）
- ⚠️ 需要维护庞大的命令白名单
- ⚠️ 无法完全防止逃逸

### 方案 3：自定义 Shell 包装器

#### 技术原理
创建一个自定义的 shell 包装器，替换真实的 bash，所有命令都经过这个包装器处理。

#### 实现方式
```typescript
// secure-shell.ts
#!/usr/bin/env bun

class SecureShell {
  private userRoot: string;
  private userId: string;
  
  constructor(userId: string) {
    this.userId = userId;
    this.userRoot = `/app/workspaces/users/${userId}`;
    process.chdir(this.userRoot);
  }
  
  async prompt(): Promise<void> {
    // 显示虚拟提示符，不暴露真实路径
    const prompt = `${this.userId}$ `;
    process.stdout.write(prompt);
  }
  
  async handleCommand(cmd: string): Promise<void> {
    // 特殊处理 cd 命令
    if (cmd.startsWith('cd ')) {
      await this.handleCd(cmd);
      return;
    }
    
    // 特殊处理 pwd 命令
    if (cmd === 'pwd') {
      process.stdout.write(`/${this.userId}\n`);
      return;
    }
    
    // 其他命令通过验证后执行
    // ...
  }
  
  async handleCd(cmd: string): Promise<void> {
    const target = cmd.slice(3).trim();
    const newPath = path.resolve(process.cwd(), target);
    
    // 验证目标路径
    if (!newPath.startsWith(this.userRoot)) {
      process.stderr.write('cd: 禁止访问该目录\n');
      return;
    }
    
    process.chdir(newPath);
  }
}
```

#### 优点
- ✅ 完全控制命令执行
- ✅ 可以自定义提示符和路径显示
- ✅ 实现相对简单

#### 缺点
- ⚠️ 需要重新实现大量 shell 内置命令
- ⚠️ 兼容性问题（某些命令行为可能不一致）

### 方案 4：容器化隔离（Docker/LXC）

#### 技术原理
为每个用户创建一个轻量级容器，在容器内运行终端。

#### 实现方式
```typescript
import { spawn } from 'child_process';

// 使用 Docker 运行隔离的终端
const container = spawn('docker', [
  'run',
  '--rm',
  '--interactive',
  '--tty',
  '--volume', `${userWorkspace}:/home/user`,
  '--workdir', '/home/user',
  '--cap-drop=ALL',  // 丢弃所有能力
  '--read-only',     // 只读文件系统（可选）
  'alpine:latest',
  '/bin/sh'
]);
```

#### 优点
- ✅ 最强隔离级别
- ✅ 成熟的容器技术
- ✅ 易于管理和清理

#### 缺点
- ⚠️ 资源开销大
- ⚠️ 启动速度慢
- ⚠️ 需要 Docker 环境

## 三、推荐方案：混合隔离架构

### 架构设计

```
┌─────────────────────────────────────────┐
│          Web 前端终端界面                │
│  (显示：user-id$ 而不是真实路径)          │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│        命令验证中间层                     │
│  - 路径合法性检查                         │
│  - 命令白名单过滤                         │
│  - 危险操作拦截                           │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Linux Namespaces 隔离层 (仅 Linux)     │
│  - Mount Namespace 文件系统隔离          │
│  - User Namespace 权限隔离               │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│    Bun PTY + Shell 执行层               │
│  - 真实的 bash 进程                      │
│  - 受限的执行环境                         │
└─────────────────────────────────────────┘
```

### 核心实现模块

#### 1. 路径验证模块
```typescript
// src/security/path-sandbox.ts

export class PathSandbox {
  private userRoot: string;
  private currentPath: string;
  
  constructor(userId: string) {
    this.userRoot = path.resolve('/app/workspaces/users', userId);
    this.currentPath = this.userRoot;
  }
  
  /**
   * 验证并解析路径，确保在用户根目录内
   */
  resolvePath(targetPath: string): string | null {
    // 处理相对路径和绝对路径
    const resolved = path.isAbsolute(targetPath)
      ? path.resolve(targetPath)
      : path.resolve(this.currentPath, targetPath);
    
    // 规范化路径（消除 .. 和 .）
    const normalized = path.normalize(resolved);
    
    // 检查是否在用户根目录内
    if (!normalized.startsWith(this.userRoot)) {
      return null;
    }
    
    return normalized;
  }
  
  /**
   * 检查命令是否允许执行
   */
  isCommandAllowed(cmd: string): boolean {
    const dangerousCommands = [
      'sudo', 'su', 'chmod', 'chown',
      'mount', 'umount', 'dd',
      'curl', 'wget', 'nc', 'netcat',
      'ssh', 'scp', 'rsync'
    ];
    
    const baseCmd = path.basename(cmd.split(/\s+/)[0]);
    return !dangerousCommands.includes(baseCmd);
  }
  
  /**
   * 获取虚拟路径显示（隐藏真实路径）
   */
  getVirtualPath(): string {
    const relativePath = path.relative(this.userRoot, this.currentPath);
    return `/${relativePath || ''}`;
  }
}
```

#### 2. 安全终端包装器
```typescript
// src/security/secure-terminal.ts

import { spawn } from 'bun-pty-rust';
import { PathSandbox } from './path-sandbox';

export class SecureTerminal {
  private pty: IPty | null = null;
  private sandbox: PathSandbox;
  private userId: string;
  private commandBuffer: string = '';
  
  constructor(userId: string) {
    this.userId = userId;
    this.sandbox = new PathSandbox(userId);
  }
  
  async start(): Promise<void> {
    // 方案 A：Linux 环境下使用 namespace 隔离
    if (process.platform === 'linux') {
      await this.startWithNamespace();
    } else {
      // 方案 B：其他平台使用命令验证层
      await this.startWithValidation();
    }
  }
  
  private async startWithNamespace(): Promise<void> {
    // 创建隔离的 namespace
    this.pty = spawn('unshare', [
      '--mount',
      '--user',
      '--map-root-user',
      '--fork',
      'bash',
      '--norc',
      '--noprofile'
    ], {
      cwd: this.sandbox.getUserRoot(),
      env: {
        ...process.env,
        HOME: this.sandbox.getUserRoot(),
        USER_ID: this.userId,
      },
      name: 'xterm-256color',
      cols: 80,
      rows: 24,
    });
    
    this.setupEventHandlers();
  }
  
  private async startWithValidation(): Promise<void> {
    this.pty = spawn('bash', ['--norc', '--noprofile'], {
      cwd: this.sandbox.getUserRoot(),
      env: {
        ...process.env,
        HOME: this.sandbox.getUserRoot(),
        PROMPT: `${this.userId}$ `,
      },
      name: 'xterm-256color',
      cols: 80,
      rows: 24,
    });
    
    this.setupEventHandlers();
  }
  
  private setupEventHandlers(): void {
    this.pty?.onData((data) => {
      // 转发输出到前端
      this.forwardToClient(data);
    });
    
    this.pty?.onData((input) => {
      // 拦截输入，验证命令
      this.commandBuffer += input;
      
      // 检测命令结束（回车）
      if (input.includes('\n')) {
        const command = this.commandBuffer.trim();
        this.commandBuffer = '';
        
        // 验证命令
        if (!this.validateCommand(command)) {
          this.pty?.write('\r\n命令被拒绝：超出访问权限\r\n');
          this.pty?.write(`${this.userId}$ `);
          return;
        }
      }
      
      // 转发到 PTY
      this.pty?.write(input);
    });
  }
  
  private validateCommand(cmd: string): boolean {
    // 提取命令（忽略前面的路径）
    const parts = cmd.match(/(?:^|[;&|])\s*([^\s;&|]+)/g);
    if (!parts) return true;
    
    for (const part of parts) {
      const command = part.trim();
      if (!command) continue;
      
      // 检查命令白名单
      if (!this.sandbox.isCommandAllowed(command)) {
        return false;
      }
      
      // 检查路径参数
      const args = command.split(/\s+/);
      for (let i = 1; i < args.length; i++) {
        const arg = args[i];
        if (arg.startsWith('-')) continue; // 跳过选项
        
        const resolved = this.sandbox.resolvePath(arg);
        if (resolved === null) {
          return false;
        }
      }
    }
    
    return true;
  }
  
  private forwardToClient(data: string): void {
    // 替换路径显示，隐藏真实路径
    const virtualData = data.replace(
      new RegExp(this.sandbox.getUserRoot(), 'g'),
      `/${this.userId}`
    );
    
    // 发送到前端
    // ...
  }
}
```

function sanitizePath(inputPath: string): string | null {
  // 1. 规范化路径
  const normalized = path.normalize(inputPath);
  
  // 2. 解析所有符号链接
  const resolved = path.resolve(normalized);
  
  // 3. 检查是否包含 ..
  if (inputPath.includes('..')) {
    return null;
  }
  
  // 4. 检查是否在允许范围内

  }
  
  return resolved;
}

## 七、风险评估

### 已知风险
1. **Namespace 逃逸**：需要 root 权限，配置不当可能被逃逸
2. **命令注入**：白名单维护不完整可能导致绕过
3. **符号链接攻击**：通过符号链接可能访问外部文件
4. **资源耗尽**：用户可能创建大量文件占用磁盘空间

### 缓解措施
1. 定期更新和审查白名单
2. 禁用符号链接或严格验证
3. 实施配额管理（磁盘、CPU、内存）
4. 启用详细的审计日志
5. 定期进行安全测试

## 九、总结

**推荐采用混合隔离方案**：
- **Linux 环境**：Namespaces + 命令验证层 + 路径沙箱（三层防护）
- **Windows 环境**：命令验证层 + 路径沙箱（双层防护）

这种方案在保证安全性的同时，兼顾了跨平台兼容性和实现复杂度。
